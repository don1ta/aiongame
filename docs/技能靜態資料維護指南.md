# 技能靜態資料維護指南

> **文件目的**：說明 `js/skill_data_static.js` 的來源、運作原理，以及日後更新技能資料的標準流程。

---

## 為什麼需要靜態資料？

查詢角色時，頁面需要顯示每個技能的**說明文字與數值**。若直接即時打 QuestLog API，會有以下問題：

| 問題 | 說明 |
|------|------|
| 🐢 速度慢 | 每個技能都需要等待一次 API 回應 |
| 🔁 重複請求 | 每次重新整理都會重打同樣的 API |
| 🚫 可能被限速 | API 服務若偵測大量請求，可能封鎖 IP |
| 📡 依賴外部服務 | 若 QuestLog API 掛掉，技能資料就空白了 |

**解決方式**：預先將所有技能資料爬下來存成靜態 JS 檔案，網頁直接讀本地資料，完全不打 API。

---

## 檔案說明

| 檔案 | 說明 |
|------|------|
| `js/skill-names.js` | 技能 ID ↔ 技能名稱的對照表（手動維護） |
| `js/skill_data_static.js` | 全量技能靜態資料（**由腳本自動生成，勿手動修改**） |
| `fetch_skills.js` | 抓取腳本，讀取 `skill-names.js` 並生成靜態資料 |
| `js/skill-api.js` | 前端讀取介面，優先讀靜態資料，找不到才呼叫 API |

---

## 運作流程

```
本機執行 node fetch_skills.js
          │
          ▼
  讀取 js/skill-names.js (272個技能ID)
          │
          ▼
  逐一請求 QuestLog API 抓取技能資料
  （僅此步驟會打 API，而且只在本機執行）
          │
          ▼
  生成 js/skill_data_static.js
          │
          ▼
  git commit + git push
          │
          ▼
  GitHub Pages 更新
  使用者查詢角色 → 讀靜態資料 → 0 次 API 請求
```

---

## 日常更新流程

### 情況一：遊戲新增了技能

1. **在 `js/skill-names.js` 加入新技能的 ID 與名稱**

```javascript
// 範例：在對應職業的區塊新增
"11810000": "新技能名稱",
```

2. **執行抓取腳本**

```powershell
node fetch_skills.js
```

腳本會自動掃描 `skill-names.js` 的全部 ID，抓取後覆蓋更新 `js/skill_data_static.js`。

3. **推送到 GitHub**

```powershell
git add js/skill_data_static.js
git commit -m "chore: 更新技能靜態資料庫"
git push
```

---

### 情況二：技能描述文字在遊戲更新後改變

同樣重新執行步驟 2、3 即可。腳本每次都是全量重抓，不需要指定哪個技能。

---

## 快取機制說明

即使在靜態資料庫中找不到某個技能（例如遊戲剛更新還沒收錄），`skill-api.js` 會自動 fallback 去打 QuestLog API。

抓回來的資料會存入 `localStorage`，有效期為 **24 小時**：

```javascript
const CACHE_EXPIRE = 24 * 60 * 60 * 1000; // 24 小時
```

所以即使靜態資料沒有的技能，同一位使用者在 24 小時內也只會打一次 API。

---

## 常見問題

### Q：`node fetch_skills.js` 跑完有些技能是 ❌，怎麼辦？

這代表 QuestLog 資料庫還沒有該技能的資料（可能遊戲剛更新），屬正常情況。  
腳本已加入「失敗自動重試 3 次」機制，若 3 次都失敗，表示 API 該技能確實無資料。  
待未來 QuestLog 補上後，重新跑腳本即可。

### Q：每次 `git push` 都需要重跑腳本嗎？

**不需要。** 只有在「需要更新技能資料」時才跑腳本。普通的功能更新、UI 修改直接 push 即可，靜態資料不會受影響。

### Q：`js/skill_data_static.js` 很大，有辦法縮小嗎？

可以，但目前約 1MB 的大小對 GitHub Pages 來說沒有顯著影響，且瀏覽器只需載入一次後會快取。暫時不需要特別優化。
